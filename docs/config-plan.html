Config Plan

Agents resolve automatically by default — bttrai stores no agent list. Users who want a fixed set can pin one in the config.

Part 3 of 4. The config file also holds presets, but preset behavior lives in plan 4. This plan owns the file itself, agent resolution, and bttrai config.

Goal

Don't make users manage agents. By default (auto mode) bttrai resolves agent targets at runtime and persists nothing about agents. A user who wants a fixed set defines agents and sets autoAgents: false.

bttrai detect                  # auto mode: resolve agents automatically
bttrai detect --agent cursor   # override agents for this run
bttrai config                  # open the file to pin agents / edit presets

Decisions

  • autoAgents (default true) means agent targets are resolved automatically at runtime; nothing is stored.
  • To pin agents, set autoAgents: false and list canonical names in agents.
  • --agent overrides agents for that run, regardless of mode.
  • bttrai never writes the agents field — it is user territory.
  • Agents use canonical bttrai names; runtime validation uses Zod.

Config Shape

Use JSON. By default agents resolve automatically and nothing is stored; set autoAgents: false and list agents to pin a fixed set. The presets block is documented in plan 4; it lives in the same file.

{
  "version": 1,                                    // schema version, for migrations
  "autoAgents": true,                              // resolve agents automatically
  "agents": [],                                    // user-defined, used only when autoAgents:false

  "presets": {
    "frontend": { "...": "see preset plan" }
  }
}
  • autoAgents: true — agents is ignored; targets are resolved at runtime.
  • autoAgents: false — agents is the fixed list; bttrai reads it and never writes it.

Add $schema later only if there is a hosted or packaged local schema.

Config Validation

Validate the config on every run before using it. Never silently fall back on a broken file — a breaking or invalid config stops the run and prompts the user to fix it.

  • When: on load, at the start of any command that reads config.
  • Checks: JSON parses, shape matches the Zod schema, agents are known canonical names, preset mcp/skills resolve to real registry entries.
  • Breaking version: if version is newer/older than supported, treat as breaking; offer a safe auto-migration when possible, otherwise prompt to fix.
  • On failure (TTY): print the exact errors + file path, then prompt — Open config to fix / Reset to defaults / Abort.
  • On failure (non-TTY or --json): emit structured errors and exit non-zero; never auto-mutate the file.

Agent Resolution

resolve:
  --agent                              (always wins for the run)
  else autoAgents ? auto-detect        (resolve targets at runtime)
                  : config.agents      (user-defined, fixed)
  else prompt                          (fallback)

no write-back: bttrai never persists agents

Auto mode detects agent targets at runtime (e.g. from agent config dirs present in the project plus built-in defaults). Resolved agents replace defaults — do not merge. bttrai never writes the agents field.

Update the detect flow handler (the service repurposed from InstallService in plan 1) so it resolves agents per the rule above. The current --auto requires --agent error goes away since auto mode always resolves something.

bttrai config Command

Optional convenience — the file is normally created by the first install/detect run, but this lets users inspect or tweak it directly.

  • Resolve the config path for the current OS.
  • If missing, write an empty config (autoAgents: true, empty agents, empty presets).
  • Open it in the user's default code editor: prefer $VISUAL / $EDITOR, else fall back to the OS default handler (open on macOS, xdg-open on Linux, start on Windows).
  • In non-TTY/JSON mode, skip launching and print the path instead.

Implementation Steps

  1. Add the canonical agent translation map.
  2. Add a Zod config schema, a JSON loader, and a writer.
  3. Add a validateConfig step that runs on load: schema + semantic checks + version compatibility, returning typed errors.
  4. On invalid config, prompt to fix (open / reset / abort) in TTY; exit non-zero with errors otherwise.
  5. In the detect flow, resolve agents as --agent → (autoAgents ? auto-detect : config.agents) → prompt; never persist agents.
  6. Add the bttrai config command to open (and lazily create) the file.
  7. Update README.
  8. Add unit tests for load/validate, agent resolution per mode, and agent translation.

Validation Errors

  • Malformed JSON.
  • Invalid config shape (schema mismatch).
  • Unknown agent name.
  • Preset references a missing MCP server or skill.
  • Unsupported / breaking version.

Tests

  • With autoAgents: true, agents resolve automatically and agents is ignored.
  • With autoAgents: false, the agents list is used as-is.
  • bttrai never writes the agents field (even with --agent).
  • --agent overrides agents for the run in either mode.
  • Parses JSON config and rejects an invalid shape.
  • Invalid config prompts to fix in TTY and exits non-zero (with errors) in non-TTY/--json.
  • Unknown agent, bad preset reference, and unsupported version each fail validation.
  • Translates canonical agents into MCP and skill targets.
  • Opens or prints config path from bttrai config.
