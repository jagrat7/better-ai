Dynamic Skill Discovery Plan

Find skills for packages that are not in the static registry, so bttrai install <pkg> works for any package — not just the hand-curated list in src/registry/skills.ts.

Problem

Today skill resolution is a closed registry. matchSkills only iterates the static skills array filtered by when.deps. A package not listed there matches nothing, and the package install prints "No extras found". The org list in src/registry/orgs.ts (OFFICIAL_ORGS) is never imported — it does not drive any lookup.

Repo names are not derivable from the package: neondatabase/agent-skills vs clerk/skills vs mapbox/mapbox-agent-skills. And the npm scope is not always the GitHub org (ai → vercel-labs). So resolution must be done live, not computed.

Goal

For deps unmatched by the static registry, resolve a GitHub repo at runtime and reuse the existing discoverRepoSkills tree-scan on it. The static registry stays the fast, authoritative first pass; this is pure fallback.

Resolution Order

Per dep, stop at the first strategy that yields a repo containing SKILL.md:

#StrategySignalWins for
0Static registrywhen.deps matchcurated packages
1npm repo fieldregistry.npmjs.org/<pkg> → repository.urlskills in main repo (ai → vercel/ai)
2GitHub repo search/search/repositories?q=<pkg> skillseparate skill repos (hono → yusukebe/hono-skill)

Flow

1
Static match first — existing matchSkills path is unchanged; covered deps never hit the network beyond their tree scan.
2
Compute unmatched deps — deps not present in any matched entry's when.deps.
3
npm repo lookup — read repository.url, parse owner/repo, run discoverRepoSkills; keep if it returns skills.
4
GitHub search fallback — top few repo hits, discoverRepoSkills each, keep the first with skills.
5
Emit as github-detected entries — append to matchSkills results with detectionSource: "github".

Gating: package-install only

The detect flow scans every project dep (dozens). Probing all of them would trip GitHub's unauthenticated search limit (~10 req/min) immediately. So discovery is opt-in via a new discover?: boolean on MatcherInput, set only by package-install, where the deps are the few packages the user explicitly named.

Method Sketch

resolveNpmRepo(pkg)        -> "owner/repo" | null   // repository.url, parse github
searchSkillRepos(pkg)      -> string[]              // /search/repositories hits
resolveDynamicSource(dep)  -> { source, skills } | null
  // 1. npm repo -> discoverRepoSkills, keep if non-empty
  // 2. else each search hit -> discoverRepoSkills, keep first non-empty

Edge Cases

  • Rate limits: unauthenticated search is ~10/min. Gating to named packages keeps calls to 1–3 per install. Honor a non-OK response by returning [] (silent fallback), same as getRepoSkillPaths.
  • Scoped names: npm URL takes @scope/name as-is; search query uses the bare last segment to cut noise.
  • repository.url variants: handle git+https://…/repo.git, git://, git@github.com:owner/repo — single regex, github-only.
  • Dedup: two deps resolving to the same repo collapse to one entry; reuse the name-dedup already added to discoverRepoSkills.
  • No progress phase: package-install calls run() without onProgress, so no spinner/UI change needed.

Implementation Steps

  1. Add discover?: boolean to MatcherInput in matcher/types.ts.
  2. Add resolveNpmRepo, searchSkillRepos, resolveDynamicSource to matcherService.
  3. In matchSkills, after static results, when discover is set: compute unmatched deps, resolve each, append github-detected entries (deduped by source).
  4. Thread discover from run() into matchSkills.
  5. Pass discover: true from package-install.ts.

Tests

  • resolveNpmRepo parses github from git+https, git@, and bare https forms; returns null for non-github.
  • npm-repo path wins when the main repo has SKILL.md (ai → vercel/ai).
  • search fallback fires when the main repo has no skills (hono → yusukebe/hono-skill).
  • discover: false (detect flow) performs no npm/search calls.
  • Unmatched dep with no resolvable repo yields no entry and no throw.
