Deeper TanStack Intent Integration

Consume the open Agent Skills convention TanStack Intent popularised — skills shipped inside the npm tarball — starting with a local node_modules scan (Stage 0), then a roadmap toward honouring the intent manifest field, requires ordering, and the trust allowlist.

Problem

Every existing discovery stage resolves skills off GitHub — the Search API (10/min unauth) caps how many deps we can probe, and a tree-scan reads the repo's HEAD, which can drift from the version actually installed. Meanwhile, packages that adopt TanStack Intent already ship their SKILL.md files in the tarball, sitting in node_modules right now — free, offline, and version-pinned. We were ignoring the cheapest, most authoritative source.

How Intent ships skills

Verified against TanStack's own scanner (packages/intent/src/discovery/scanner.ts):

  • Skills live at <pkg>/skills/<name>/SKILL.md, published in the npm tarball, so they land in node_modules/<dep>/skills/ on install. A SKILL.md directly in skills/ root is ignored — a skill needs its own folder.
  • A package signals support via an intent manifest field ({ version: 1, repo, docs, requires?[] }), or it is derived from a skills/ dir + a repository field.
  • The skill name is the folder path relative to skills/; frontmatter carries description, type, framework. Same open SKILL.md spec bttrai already parses.
  • intent.requires → cross-package skill ordering (topological). intent.skills in the consumer manifest is a trust allowlist (* = allow-all, [] = deny-all, absent = show-all).

Stage 0 — Local node_modules scan build now

For each dep, walk node_modules/<dep>/skills/**/SKILL.md on the local filesystem and parse each file. No network, no rate limit, pinned to the installed version. Runs for every flow that knows the project root — both detect (which the GitHub stages are too rate-limited to serve) and install <pkg> — ahead of the GitHub stages. Deps resolved here are marked covered so later stages skip them.

Stage 0 Flow

1
Resolve path — join(project, "node_modules", dep, "skills") per dep, in parallel.
2
Walk for SKILL.md — readdir(skillsDir, { recursive: true }); keep nested */SKILL.md only. Missing dir → empty, package isn't Intent-bearing.
3
Parse name — frontmatter name: slug if present, else the folder path (Intent's convention). Shared parseSkillName with the GitHub path.
4
Emit + cover — append as detectionSource: "local"; record the dep in coveredDeps so stages 1–3 skip it.

Roadmap — deeper integration later

  • Honour the intent field: read repo/docs and frontmatter type/framework for richer display, instead of relying on the skills/ dir alone.
  • requires ordering: topologically sort installed skills by intent.requires so dependent skills install in order.
  • Trust allowlist: respect a consumer's intent.skills allowlist (* / [] / explicit) before wiring anything in.
  • Registry consumption: query the TanStack Intent registry (npm keyword tanstack-intent) as a network tier for deps that ship skills out-of-tree — complements, doesn't replace, Stage 0.

Edge Cases

  • No skills/ dir / no node_modules: readdir throws → caught → empty, falls through to GitHub/registry. Same silent-fallback contract as getRepoSkillPaths.
  • Top-level skills/SKILL.md: ignored — only nested */SKILL.md count, matching Intent.
  • Duplicate skill names across folders: first-wins via dedupByName.
  • Monorepo workspaces / PnP: out of scope for Stage 0 — plain node_modules only; workspace + Yarn PnP roots are roadmap material.
