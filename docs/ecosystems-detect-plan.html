Ecosyste.ms-Backed Detect Discovery

Extend dynamic skill discovery (plan 5) to the full detect scan by moving the package→repo lookup off GitHub's quota onto the free ecosyste.ms API, so every project dep can be probed without tripping rate limits.

Problem

Plan 5 gates discovery to package-install only. detect scans every dep (50–150 in a real project), and the discovery tiers lean on the GitHub Search API — capped at 10/min unauthenticated, 30/min with a token. Two searches per dep × 100 deps is instant rate-limit death, or ~7 minutes of throttled waiting even authenticated. The tree-scan (core REST, 5,000/hr authed) is not the bottleneck — the Search calls are.

Goal

Make full discovery viable on detect by replacing the per-dep GitHub lookup with a non-GitHub source whose cost does not scale with the Search quota, and caching results so repeat runs are free. The static registry stays the offline-safe fallback.

Why ecosyste.ms

packages.ecosyste.ms already aggregates npm → repository → repo metadata from the GH Archive firehose. One call to a non-GitHub host returns both the repo URL and its topics — replacing tier 1a (resolveNpmRepo) and the owner/global search tiers in a single request that never touches GitHub's quota. 

GET packages.ecosyste.ms/api/v1/registries/npmjs.org/packages/<pkg>
  -> repository_url         // "https://github.com/honojs/hono"
  -> repo_metadata.topics   // ["web-framework", "agent-skills", ...]

Resolution Order (detect)

# Source Cost Notes
1 ecosyste.ms package lookup 1 call / dep, off GitHub quota repo URL + topics; skip if cached
2 GitHub tree-scan of the resolved repo core REST (5,000/hr), ETag-cached existing discoverRepoSkills; only for repos that look skill-bearing
3 Static registry offline fallback when ecosyste.ms is unreachable

Flow

1
Dedup deps by owner — many deps share a repo (@vercel/*); collapse before any network call.
2
Cache check — look up dep → repo → skills in a local TTL cache; serve hits with zero network.
3
ecosyste.ms resolve — fetch repo URL + topics; keep only repos whose topics/name look skill-bearing (agent-skills, *skill*).
4
GitHub tree-scan (ETag) — discoverRepoSkills on survivors, sending If-None-Match; a 304 costs no rate limit.
5
Persist + emit — write the cache entry, append as detectionSource: "github", dedup by source.

Rate-Limit Mechanics

Lever Effect
ecosyste.ms for repo lookup removes the per-dep GitHub Search calls entirely (the real cap)
Owner dedup 100 deps → ~20–30 unique repos before any fetch
Topic pre-filter only tree-scan repos that advertise skills; skip the rest for free
ETag conditional requests 304 responses do not count against the GitHub limit
Local TTL cache first detect pays once; reruns and overlapping projects are free

Method Sketch

resolveEcosystemsRepo(pkg) -> { repo, topics } | null   // packages.ecosyste.ms, no GitHub quota
looksSkillBearing(repo, topics) -> boolean              // "agent-skills" topic | *skill* name
discoverRepoSkills(repo, etag?) -> { skills, etag }     // If-None-Match -> 304 = free
cacheGet(dep) / cacheSet(dep, entry)                    // TTL JSON under ~/.cache/bttrai

Edge Cases

  • ecosyste.ms down/slow: any non-OK or timeout → fall back to the static registry (offline-safe), same silent-fallback contract as getRepoSkillPaths.
  • Stale cache: TTL (e.g. 24h) bounds staleness; a registry bump or --no-cache forces a refresh.
  • Topic gaps: repos that ship skills without the topic still get caught by the existing *skills* name heuristic before tree-scan.
  • Monorepos: ecosyste.ms may point many packages at one repo — owner/source dedup already collapses these to a single tree-scan.

Implementation Steps

  1. Add resolveEcosystemsRepo to matcher/utils.ts.
  2. Add ETag support to the tree fetch; thread the ETag through the disk cache.
  3. Add TTL cache read/write helpers in shared/utils.ts.
  4. In stage 1, dedup by owner → cache check → ecosyste.ms resolve → topic filter → tree-scan.
  5. Enable discover: true on the detect flow.
  6. Keep package-install on the same path (it just probes fewer deps).
